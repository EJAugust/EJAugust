I find metaphors to be a useful way to develop and present new data structures. The file folder metaphor is perhaps one of the most obvious examples of this utility. There are, of course, no paper files or folders inside the computer’s case.

In this note, I explore the multiverse as a metaphor for a highly dynamic (but currently hypothetical) web technology I’m designing. This metaphor helps me keep two kinds of data separate in my mind.

Two Kinds of Change

Users are used to interacting with a web app without those interactions changing the whole experience in fundamental ways. Typically, users are familiar with interactions like browsing to different pages and changing basic settings.

They are also used to seeing the current page location reflected in their address bar, thanks to a routing system.

What sets our hypothetical project apart, however, is that the user can also modify the source code of the web app itself, right in their browser.

If the source code of the app is stored in the address bar (or, conversely, if the page you’re on could only be changed by editing the source code) we would lose the separation between browsing and software engineering.

This would be a massive technical challenge. The URI might have to embed a significant amount of code. A note I wrote last week hints at how this might be achieved - if it can be achieved at all.

In addition to this problem, depending on what changes they make, the user could crash the app or make it impossible to use. The lack of distinction between safe and normal browsing activity and doing surgery to the application might confuse users.

That’s where the multiverse metaphor comes in…

Which Universe Am I In?

We can split all modifications to the source code into two distinct categories:

  1. Which universe am I in?

  2. Where am I within that universe?

My hope is that this division will be helpful in explaining to users the difference between normal routing and source code changes (and hint at some of the consequences of editing an app while its still running).

An empty document is just an empty document. The behavior of the web app doesn’t appear until you add the source code. When you change that source code from within the app itself, you are basically changing the “laws of physics” for that web app universe.

If all of the web app’s source code can be modified by the user, then they can easily destroy the universe. They can also improve it. The application, however, is static (as static as the blank document). So, the universe can never change. Instead, the user is traveling the multiverse - moving from one universe to the next.

Luckily, in this case, we’re just talking about a web application and not the actual universe. However, you never know …

In a safe, reliable universe, you should be able to move anywhere you want without destroying that universe (I think that’s reasonable).

Nobody wants to accidentally change the laws of physics. This metaphor creates the illusion of two kinds of change - intra-universal change and inter-universal change - when in fact their is only one - the user’s current address.

Conclusion

My dream is to make software engineering easy and accessible to everyone. Whether it’s real or not, the multiverse may be able to serve as a useful metaphor for introducing new experiences to users.

This conversation is one of many ongoing threads that I’m exploring. Time will tell which of them pans out.
